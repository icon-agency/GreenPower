# LESSONS.md

Running log of corrections. Each entry prevents a future mistake.
Read before starting work. Append (don't edit) when something goes wrong.

- `text-wrap: pretty` is the site-wide default (set on `body`, inherits to every text node including headings). The hero h1 (`.hero__title`) is the lone exception, kept on `text-wrap: balance` for its display headline.
- Wire text wrapping into base typography or a shared mixin. Do not re-apply it per component.
- Raster images must ship with `srcset` + `sizes`. The page scales fluidly above the `wide` breakpoint (clamp on root font-size, container max grows to 1600px), so single-resolution images will look soft on large displays.
- Hero (and any rounded, full-bleed panel with a GPU-composited pseudo-element): use `overflow: hidden`, not `overflow: clip`. `overflow: clip` skips the Block Formatting Context so the compositor is not guaranteed to clip the promoted layer flush with the edge during resize repaints — a sub-pixel strip of the gradient background bleeds through on the right. Also give the `::before` a small horizontal overbleed (`inset: … -2px`) as insurance against fractional-pixel hero widths.
- Background parallax layers (`data-parallax-bg`) must compute their viewport offset from `el.getBoundingClientRect()` directly — **never** via `el.offsetParent`. The hero section has no positioned ancestor, so `offsetParent` resolves to `<body>` (full page height ~4800 px), producing an offset ~20× too large that overshoots the parallax buffer and exposes the gradient background at the bottom of the hero. Foreground layers (`data-parallax`) are `position: absolute` inside a positioned container, so `offsetParent` is correct for those.
- Cropped image parallax needs a height-based bleed buffer. If the image itself moves inside a rounded/hidden container, extend it beyond the mask before translating so card or panel background cannot show at the crop edge.
- Enter animation classes must stay applied for the full keyframe duration. Removing an `is-entering` class on the next `requestAnimationFrame` collapses a fade/scale/slide animation into a barely visible snap.
- Don't rely on `order` to move children of a `display: contents` wrapper across flex lines in a wrapping container — Chrome's line-packing goes haywire (items land mid-line, overlapping other rows). Give the wrapper a real box (`display: block`) and order the wrapper itself instead.
- 3D-flip wrappers around content containing absolutely-positioned descendants (mega panels) silently re-anchor them: `transform`, `perspective`, `position: relative`, and (in Chrome) `transform-style: preserve-3d` on an ancestor all create a containing block. Gate every one of those properties behind an "actively animating" state class (`.is-flipping`) so at rest the wrapper is inert and panels keep anchoring to the header.
- Don't build underlines with `box-shadow: 0 2px 0 0`. When the element sits at a fractional x-position (fluid layouts guarantee this), the shadow knockout leaves a 1px vertical seam at the element's left edge. Use an absolutely-positioned `::after` bar instead — it also allows independent hover/active underline colours.
- The `flex-row()` mixin must set `flex-direction: row` explicitly. In mobile-first components the base styles apply `flex-col()` (column), and a desktop `flex-row()` override that only sets `display: flex` inherits the column direction — the nav menu rendered stacked at desktop until the mixin reset the direction.
- Percentage-height decorative pseudos on the header track content-driven growth. The overlay nav's `::before` scrim (`height: 200%`) was sized against the idle bar, but under `no-js` the fallback menu expands *inside* the header — the scrim computed to 5146px, veiled the whole page in 35% black + blur, and its overflow added ~3900px of phantom document scroll. Scope such overlays to `.js` (like the rest of the overlay machinery) or derive their size from a fixed value, never from a container whose height any state can grow.
- Scroll-linked JS transforms (`scroll` → rAF → `style.transform`) visibly lag native scrolling — browsers scroll on the compositor thread, so the JS write lands a frame or more late and the element "swims" against content that scrolls natively. To lock an element to the page/another scrolling element, put it IN the scrolling layer (e.g. an absolute full-viewport wrapper anchored to the container's first 100svh, like `.hero__follow`/`.hero__content`) and let the browser move it. A sticky sibling + scroll-layer sibling also reproduce "move together while pinned, freeze after" for free — the JS emulation (`heroTop - frameTop`) is only needed for non-1:1 factors like parallax.
- A scrim/overlay that must reach the BOTTOM of a taller-than-viewport section must anchor to that section (`bottom: 0` inside `.hero`), not to a 100svh companion layer. The hero's `__follow` layer is only 100svh, but the shape panel overhangs the first viewport, so its base IS the hero's base — below 100svh. The `.hero__blur` scrim, pinned to the follow layer's bottom, therefore stopped ~220px short of the hero base below the nav breakpoint, exposing a strip of un-blurred video (desktop hid it: the follow layer is `sticky`, so its bottom rides down to the hero base at the end of the journey). Anchoring the blur to `.hero` with `top: 46svh; bottom: 0` fixes every regime, and because the base sits below the viewport until the hero scrolls off, the desktop pinned-to-viewport-bottom look is preserved for free. Use a length-based mask ramp (`black 31svh`, not `58%`) so "opaque by ~77svh" holds regardless of the now-variable band height.
